# Plan

## Domain

A reading list ‚Äî deliberately the simplest domain on offer: one entity, no relations, no date math, no money handling.

### Entity: `book`

| Field       | Type           | Rules                                   |
| ----------- | -------------- | --------------------------------------- |
| `id`        | UUID           | Generated by Postgres                   |
| `title`     | string         | Required, trimmed, 1‚Äì200 chars          |
| `author`    | string \| null | Optional, trimmed, max 200 chars        |
| `status`    | enum           | `to_read` \| `reading` \| `finished`    |
| `rating`    | int \| null    | Optional, 1‚Äì5                           |
| `createdAt` | timestamp      | Set by Postgres                         |
| `updatedAt` | timestamp      | Updated on change                       |

## Architecture

- **API (`apps/api`)** ‚Äî NestJS. Uses `@supabase/supabase-js` with the service-role key, held only in API env vars. Deployed to Vercel as a serverless function.
- **Mobile (`apps/mobile`)** ‚Äî Expo. Plain `fetch` against the API through a single API-client module. No state-management library; component state is enough at this size.
- **Database** ‚Äî Supabase Postgres, one `books` table created via a checked-in `supabase/schema.sql` with DB-level constraints mirroring API validation.

### API endpoints

| Method   | Route                    | Success    | Errors                          |
| -------- | ------------------------ | ---------- | ------------------------------- |
| `GET`    | `/books?search=&status=` | `200` list | `400` invalid status filter     |
| `GET`    | `/books/:id`             | `200` item | `400` bad UUID, `404` not found |
| `POST`   | `/books`                 | `201` item | `400` invalid body              |
| `PATCH`  | `/books/:id`             | `200` item | `400`, `404`                    |
| `DELETE` | `/books/:id`             | `204`      | `400` bad UUID, `404` not found |

API implementation notes:

- Global `ValidationPipe` with `transform`, `whitelist`, and `forbidNonWhitelisted` so unknown fields are rejected, not silently dropped.
- DTOs with `class-validator`; `ParseUUIDPipe` on `:id` params.
- Supabase errors are caught in the service layer and mapped to Nest HTTP exceptions ‚Äî a bad request never takes the endpoint down and DB details never leak to clients.
- CORS enabled for the mobile app.

### Mobile screens

- **List** ‚Äî loads from the API; loading, error (with retry), and empty states; search-by-title input and status filter chips.
- **Detail** ‚Äî fetches one book; edit and delete entry points.
- **Create / Edit** ‚Äî one shared `BookForm` component; client-side validation (title required, rating range) before submit; submit disabled while in flight.
- **Delete** ‚Äî native confirmation alert before calling the API; navigates back to a refreshed list.

## Trade-offs (made up front)

- **Supabase JS client over an ORM** ‚Äî no migration tooling or schema sync to configure; one small table doesn't justify Prisma/TypeORM setup cost.
- **`fetch` + component state over React Query** ‚Äî fewer moving parts; the app has four screens and no shared server-cache needs. Optimistic updates become harder, which is acceptable (they're a stretch goal).
- **No shared types package** ‚Äî a workspace package adds build config for one entity; the mobile app declares its own `Book` type instead.
- **No auth** ‚Äî out of scope for the core; Supabase Auth is the listed stretch goal if time remains.
- **Few, targeted API tests** ‚Äî cover validation rejection, happy-path create, and 404s rather than chasing coverage.

## Stretch goals, in order of value-per-effort

1. Pull-to-refresh on the list (cheap with `FlatList`).
2. Pagination via `range()` on the Supabase query + `limit/offset` query params.
3. Optimistic delete on the list.
4. Supabase Auth.
